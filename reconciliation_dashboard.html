What This Would Get Wrong In Production
Late settlement detection assumes a strict 1-month window. In production, settlement lag can span multiple batches, weekends, or holidays — requiring a configurable tolerance window, not a binary Jan/Feb check.
Rounding detection uses a threshold of <₹1 difference. In production with millions of transactions, even ₹0.005 × 10M rows = ₹50,000 leakage. Needs per-currency decimal precision rules, not a generic float comparison.
Duplicate detection matches on Transaction ID + Amount. A real system needs fuzzy matching on timestamp + customer ID + amount to catch duplicates with different IDs (e.g. retried API calls that generated new IDs).
Orphan refund detection only checks within the same dataset. Production must cross-check against archived months — the original purchase may be from a prior period, making it a false positive here.
